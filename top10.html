Os 10 principais riscos de segurança de aplicativos da Web

A01:2021-Broken Access Control A02:2021-Cryptographic Failures A03:2021-Injection A04:2021-Insecure Design A05:2021-Security Misconfiguration A06:2021-Vulnerable and Outdated Components A07:2021-Identification and Authentication Failures A08:2021-Software and Data Integrity Failures A09:2021-Security Logging and Monitoring Failures A10:2021-Server-Side Request Forgery
  • A01:2021-Broken Access Control sobe da quinta posição; 94% dos aplicativos foram testados para alguma forma de controle de acesso quebrado. As 34 Enumerações de Fraquezas Comuns (CWEs) mapeadas para Controle de Acesso Quebrado tiveram mais ocorrências em aplicativos do que qualquer outra categoria.
  • A02:2021-Cryptographic Failures sobem uma posição para a 2ª posição, anteriormente conhecida como Exposição de Dados Sensíveis, que era um sintoma amplo e não uma causa raiz. O foco renovado aqui está nas falhas relacionadas à criptografia, que geralmente levam à exposição de dados confidenciais ou ao comprometimento do sistema.
  • A03:2021-Injection desliza para a terceira posição. 94% dos aplicativos foram testados para alguma forma de injeção, e os 33 CWEs mapeados nesta categoria têm o segundo maior número de ocorrências em aplicativos. Cross-site Scripting agora faz parte desta categoria nesta edição.
  • A04:2021-Insecure Design é uma nova categoria para 2021, com foco em riscos relacionados a falhas de projeto. Se realmente queremos “ir para a esquerda” como um setor, isso exige mais uso de modelagem de ameaças, padrões e princípios de design seguros e arquiteturas de referência.
  • A05:2021-Security Misconfiguration subiu do 6º lugar na edição anterior; 90% dos aplicativos foram testados para algum tipo de configuração incorreta. Com mais mudanças em software altamente configurável, não é surpreendente ver essa categoria subir. A antiga categoria para XML External Entities (XXE) agora faz parte desta categoria.
  • A06:2021-Vulnerable and Outdated Components foi anteriormente intitulado Using Components with Known Vulnerabilities e é o número 2 na pesquisa da comunidade Top 10, mas também tinha dados suficientes para entrar no Top 10 por meio de análise de dados. Essa categoria subiu do 9º lugar em 2017 e é um problema conhecido que lutamos para testar e avaliar o risco. É a única categoria a não ter nenhuma Vulnerabilidade e Exposições Comuns (CVEs) mapeadas para os CWEs incluídos, portanto, uma exploração padrão e pesos de impacto de 5,0 são considerados em suas pontuações.
  • A07:2021-Identification and Authentication Failures eram anteriormente a autenticação quebrada e está deslizando para baixo da segunda posição e agora inclui CWEs que estão mais relacionados a falhas de identificação. Essa categoria ainda é parte integrante do Top 10, mas a maior disponibilidade de estruturas padronizadas parece estar ajudando.
  • A08:2021-Software and Data Integrity Failures é uma nova categoria para 2021, com foco em fazer suposições relacionadas a atualizações de software, dados críticos e pipelines de CI/CD sem verificar a integridade. Um dos impactos mais ponderados dos dados do Common Vulnerability and Exposures/Common Vulnerability Scoring System (CVE/CVSS) mapeados para os 10 CWEs nesta categoria. A desserialização insegura de 2017 agora faz parte dessa categoria maior.
  • A09:2021-Security Logging and Monitoring Failures eram anteriormente Registro e monitoramento insuficientes e são adicionadas a partir da pesquisa do setor (nº 3), passando do número 10 anteriormente. Essa categoria é expandida para incluir mais tipos de falhas, é difícil de testar e não está bem representada nos dados CVE/CVSS. No entanto, falhas nesta categoria podem afetar diretamente a visibilidade, alertas de incidentes e perícia.
  • A10:2021-Server-Side Request Forgery é adicionado da pesquisa da comunidade Top 10 (#1). Os dados mostram uma taxa de incidência relativamente baixa com cobertura de teste acima da média, juntamente com classificações acima da média para potencial de exploração e impacto. Esta categoria representa o cenário em que os membros da comunidade de segurança estão nos dizendo que isso é importante, embora não esteja ilustrado nos dados neste momento.

Top 10 2020 Plano de Análise de Dados

Metas

Coletar o conjunto de dados mais abrangente relacionado às vulnerabilidades de aplicativos identificadas até o momento para permitir a análise do Top 10 e de outras pesquisas futuras. Esses dados devem vir de várias fontes; fornecedores e consultorias de segurança, recompensas por bugs, juntamente com contribuições da empresa/organização. Os dados serão normalizados para permitir a comparação de nível entre ferramentas assistidas por humanos e humanos assistidos por ferramentas.


Infraestrutura de análise

Planeje aproveitar a OWASP Azure Cloud Infrastructure para coletar, analisar e armazenar os dados contribuídos.


Contribuições

Planejamos apoiar contribuições conhecidas e pseudo-anônimas. A preferência é que as contribuições sejam conhecidas; isso ajuda imensamente na validação/qualidade/confiança dos dados enviados. Se o remetente preferir que seus dados sejam armazenados anonimamente e até mesmo enviar os dados anonimamente, eles deverão ser classificados como “não verificados” versus “verificados”.

Contribuição de dados verificada

Cenário 1: O remetente é conhecido e concordou em ser identificado como parte contribuinte.
Cenário 2: O remetente é conhecido, mas prefere não ser identificado publicamente.
Cenário 3: O remetente é conhecido, mas não deseja que ele seja registrado no conjunto de dados.

Contribuição de dados não verificada

Cenário 4: O remetente é anônimo. (Devemos apoiar?)

A análise dos dados será realizada com uma distinção cuidadosa quando os dados não verificados fizerem parte do conjunto de dados que foi analisado.


Processo de Contribuição

Existem algumas maneiras de contribuir com dados:

  1. Envie por e-mail um arquivo CSV/Excel com os conjuntos de dados para brian.glas@owasp.org
  2. Carregue um arquivo CSV/Excel para uma “pasta de contribuição” (em breve)

Exemplos de modelos podem ser encontrados no GitHub: https://github.com/OWASP/Top10/tree/master/2021/Data


Período de Contribuição

Planejamos aceitar contribuições para o novo Top 10 de maio a 30 de novembro de 2020 para dados que datam de 2017 até o momento.


Estrutura de dados

Os seguintes elementos de dados são obrigatórios ou opcionais.
Quanto mais informações forem fornecidas, mais precisa será nossa análise.
No mínimo, precisamos do período de tempo, número total de aplicativos testados no conjunto de dados e a lista de CWEs e contagens de quantos aplicativos continham esse CWE.
Se possível, forneça os metadados adicionais, porque isso nos ajudará muito a obter mais informações sobre o estado atual dos testes e vulnerabilidades.

Metadados

  • Nome do Contribuinte (org ou anon)
  • E-mail de contato do colaborador
  • Período de tempo (2019, 2018, 2017)
  • Número de aplicativos testados
  • Tipo de teste (TaH, HaT, Ferramentas)
  • Idioma principal (código)
  • Região geográfica (Global, América do Norte, UE, Ásia, outros)
  • Indústria Primária (Múltipla, Financeira, Industrial, Software, ??)
  • Se os dados contêm ou não retestes ou os mesmos aplicativos várias vezes (T/F)

Dados CWE

  • Uma lista de CWEs com contagem de aplicativos encontrados para conter esse CWE

Se possível, forneça CWEs principais nos dados, não categorias CWE.
Isso ajudará na análise, qualquer normalização/agregação feita como parte dessa análise será bem documentada.

Observação:

Se um colaborador tiver dois tipos de conjuntos de dados, um de fontes HaT e outro de TaH, é recomendável enviá-los como dois conjuntos de dados separados.
HaT = Ferramentas assistidas por humanos (maior volume/frequência, principalmente de ferramentas)
TaH = Humano assistido por ferramenta (menor volume/frequência, principalmente de testes em humanos)


Enquete

Da mesma forma que o Top Ten 2017, planejamos realizar uma pesquisa para identificar até duas categorias do Top Ten que a comunidade acredita serem importantes, mas podem ainda não estar refletidas nos dados. Planejamos realizar a pesquisa em maio ou junho de 2020 e usaremos os formulários do Google de maneira semelhante à última vez. Os CWEs na pesquisa virão de descobertas de tendências atuais, CWEs que estão fora do Top Ten em dados e outras fontes potenciais.


Processo

Em um nível alto, planejamos realizar um nível de normalização de dados; no entanto, manteremos uma versão dos dados brutos contribuídos para análise futura. Analisaremos a distribuição CWE dos conjuntos de dados e potencialmente reclassificaremos alguns CWEs para consolidá-los em buckets maiores. Documentaremos cuidadosamente todas as ações de normalização tomadas para que fique claro o que foi feito.

Planejamos calcular a probabilidade seguindo o modelo que desenvolvemos em 2017 para determinar a taxa de incidência em vez da frequência para avaliar a probabilidade de um determinado aplicativo conter pelo menos uma instância de um CWE. Isso significa que não estamos procurando a taxa de frequência (número de descobertas) em um aplicativo, mas sim o número de aplicativos que tiveram uma ou mais instâncias de um CWE. Podemos calcular a taxa de incidência com base no número total de aplicativos testados no conjunto de dados em comparação com quantos aplicativos cada CWE foi encontrado.

Além disso, desenvolveremos pontuações básicas do CWSS para os 20-30 principais CWEs e incluiremos o impacto potencial na ponderação dos 10 principais.

Além disso, gostaria de explorar insights adicionais que podem ser obtidos do conjunto de dados contribuído para ver o que mais pode ser aprendido que pode ser útil para as comunidades de segurança e desenvolvimento.
